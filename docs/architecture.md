# Architecture

This document describes how chief-of-staff is structured internally. Read the [README](../README.md) first for what it does and how to use it.

## Twelve layers, one concern each

1. **Context library** (`context/`): five Markdown files for identity, stakeholders, priorities, decision principles, and writing arc.
2. **Memory** (`memory/`): decisions log, relationships, lessons learned, weekly digests.
3. **Projects** (`projects/<slug>/`): per-initiative status, decisions, commitments, notes. Three templates ship: generic, pipeline, travel.
4. **Work queue** (`data/queue.jsonl`): JSONL store for everything in flight, with provenance, audit trail, and an `assigned_to` lock. `/am-sweep` reads it first.
5. **Permission engine** (`data/tiers.json` + `tools/permit.mjs`): tier check before any side-effect action; every decision logged to `data/permit-audit.jsonl`. Each slash command declares a routine `mode` (`read-only`, `approval-required`, `autonomous`) that caps the effective tier for that command.
6. **Conformance audits** (`tools/conform.mjs`): voice and structural rules encoded as code; high-severity hits block delivery.
7. **Typed-link graph** (`tools/build-graph.mjs`, `data/graph.json`): derived graph over stakeholders, decisions, projects, queue items. Query helpers in `tools/graph-query.mjs`.
8. **Weekly digest** (`tools/compact.mjs`, `memory/digest-YYYY-Wnn.md`): folds the last 7 days of closes and decisions into one page `/am-sweep` reads first.
9. **Event-driven hooks** (`tools/hooks/`, `tools/hooks-runner.mjs`): scheduled poller for overdue items, dormant projects, dormant warm-list stakeholders, and newsletter rollup.
10. **Inbound channels** (`/api/forward`, `/api/slack/event`): forward any email from any device, or @-mention the agent in a Slack thread and approve in-thread. Both write directly to the queue with provenance. See `FORWARD_SETUP.md` and `SLACK_SETUP.md`.
11. **Subagents and slash commands** (`.claude/`): four specialists, thirteen workflows.
12. **Regression evals** (`evals/run.mjs`): synchronous fixtures for conformance, permit, queue, graph. Exits non-zero on any fail. Runs as `npm test` so a prompt change cannot silently degrade the substrate.

A standalone Node CLI (`index.mjs`) and a local Express web UI (`server.mjs`, port 3030) wrap the agent. The UI also exposes the queue (with undo + history), tiers, permit audit, conformance audit, projects, maintenance audit, forwarding endpoint, and Slack endpoint over REST. The web UI rail renders queue items, active projects, and the voice-audit pass rate.

A first-run wizard (`npm run init`) asks 6 questions and writes initial `CLAUDE.md`, `AGENTS.md`, and the four context files. Conformance rules live in `data/conform-rules.json` (ship) with `data/conform-rules.local.json` for personal overrides (gitignored).

## File and directory map

### Top-level documents

- `CLAUDE.md`: identity, voice rules, and the Dispatch/Prep/Yours/Skip triage taxonomy. The agent reads this first.
- `AGENTS.md`: short identity pointer for non-Claude tooling.
- `USAGE.md`: daily and weekly pattern, scheduled overnight setup, failure modes.
- `MOBILE.md`: cheat sheet for capturing relationship updates, decisions, and tasks from a phone.
- `MCP_SETUP.md`: one-time Google OAuth walkthrough for the Gmail and Calendar MCP servers.
- `FORWARD_SETUP.md`, `SLACK_SETUP.md`: wiring walkthroughs for the inbound channels.

### Living context

- `context/`: `stakeholders.md`, `priorities.md`, `research_arc.md`, `career_thesis.md`, `operating_principles.md`. Treated as living documents.
- `memory/`: `decisions.md`, `relationships.md`, `learnings.md`. Rolling state.
- `projects/`: per-initiative working state. `_template/` ships with the standard shape (`status.md`, `decisions.md`, `commitments.md`, `notes.md`); copy it to start a new project.
- `tasks.md`: living checklist surfaced in the web UI.

### Data substrate

- `data/queue.jsonl`: canonical work queue. JSONL, append-only, snapshot-per-line.
- `data/queue.md`: rendered Markdown view of the open queue, regenerated by `tools/queue-md.mjs`.
- `data/tiers.json`: permission engine config. Action catalog + actor tier table.
- `data/permit-audit.jsonl`: append-only audit of every permit decision.
- `data/conform-rules.json`: conformance rule library (AI tells, flattery, email bans, thresholds). Edit to customize voice rules without touching JS. Personal overrides go in `data/conform-rules.local.json` (gitignored).
- `data/conform-audit.jsonl`: append-only log of every conform check; powers the voice-audit dashboard panel.
- `data/graph.json`: typed-link graph, regenerated by `tools/build-graph.mjs`.

### Tools

- `tools/queue.mjs`, `tools/queue-cli.mjs`, `tools/queue-md.mjs`: queue library (including `claim` and `release` for inter-agent state), CLI, Markdown renderer.
- `tools/permit.mjs`, `tools/permit-cli.mjs`: permission engine library and CLI.
- `tools/provenance.mjs`: renders the `Sources:` footer for drafts from queue item provenance.
- `tools/conform.mjs`, `tools/conform-cli.mjs`: voice and structural rule auditor.
- `tools/build-graph.mjs`, `tools/graph-query.mjs`: typed-link graph builder and query helpers.
- `tools/compact.mjs`: weekly digest generator. Writes `memory/digest-YYYY-Wnn.md` summarizing the last 7 days.
- `tools/hooks-runner.mjs`, `tools/hooks/*.mjs`: event-driven hooks (overdue items, dormant projects, dormant warm-list stakeholders, newsletter rollup).
- `tools/slack-respond.mjs`: outbound Slack helper.
- `tools/init-wizard.mjs`: interactive first-run setup.
- `tools/setup.mjs`: setup orchestrator. Runs `npm install`, fills in `.env`, runs the content wizard, runs the doctor.
- `tools/doctor.mjs`: preflight and dashboard. `--quick` skips live network calls.
- `tools/check/*.mjs`: per-service smoke tests, callable as `npm run check:<service>` and aggregated by the doctor.
- `tools/composio-connect.mjs`: interactive setup for the Composio-managed MCP path.
- `tools/demo.mjs`: seeds 8 sample queue items and a demo project for `npm run cos:demo`.
- `tools/schedule.mjs`: one-shot scheduler. OS-detected (Windows `schtasks` or Unix `crontab`), idempotent.
- `tools/sync-plugin-dirs.mjs`: keeps `commands/`+`agents/` (plugin-root) in sync with `.claude/commands/`+`.claude/agents/` (project layout).

### Plugin packaging

- `.claude-plugin/plugin.json`, `.claude-plugin/marketplace.json`: Claude Code plugin manifest and marketplace metadata.
- `commands/`, `agents/`: plugin-root copies of `.claude/commands/` and `.claude/agents/`. Kept in sync by `tools/sync-plugin-dirs.mjs`.
- `hooks/hooks.json`, `hooks/notify-session-start.mjs`: SessionStart hook that prints a one-line status when the plugin loads.

### Entry points

- `index.mjs`: standalone Node CLI.
- `server.mjs`: local Express web UI on port 3030.
- `send-to-self.mjs`: small SMTP wrapper used by `email-drafter` so drafts land in your own inbox; recipient is locked at the script level.

### Container path

- `Dockerfile`, `docker-compose.yml`, `.dockerignore`: container path. Bring your own `.env`; `data/`, `projects/`, `memory/` are volume-mounted so state persists.

### Regression evals

- `evals/run.mjs`: regression suite for the deterministic substrate (conform, permit, queue, graph). 39 fixtures. Wired as `npm test`.

### Configuration

- `.mcp.json`: MCP server config for Gmail and Google Calendar.
- `.env.example`: template for keys (Anthropic, optional Maps, SMTP for send-to-self).
- `run-email-triage.bat`, `run-calendar-prep.bat`: Windows Task Scheduler wrappers for overnight runs.

## The self-send safety pattern

The `email-drafter` subagent never emails third parties directly. It produces a drafted reply and ships it to your own inbox via `send-to-self.mjs`, which is hard-coded to use `SELF_EMAIL` as the recipient. The body of the self-email states the intended external recipient, the thread reference, and the drafted text. You read it, copy or forward, and send the real reply yourself.

This is a deliberate safety design. Even if the agent misclassifies an email or hallucinates a recipient, the worst case is a draft landing in your own inbox. No third party is contacted.

SMTP credentials come from one of two `.env` files, in this order: the project's `.env` (preferred), or `../role-radar/.env` as a fallback. For Gmail SMTP, you need an App Password (not your real Gmail password).

## Permission engine and provenance

Every queue item must carry provenance (the upstream signal it came from) as a `{type, ref, captured_at}` array. When the agent drafts anything that derives from queue items, it builds a `Sources:` footer from those arrays via `tools/provenance.mjs`.

Every side-effect action goes through `tools/permit-cli.mjs check` first. Action classes and required tiers live in `data/tiers.json`. Actor tiers also live there. Defaults are tier 0 (read, draft, write to queue, write to memory, write to logs). Higher-tier actions require an explicit `permit-cli.mjs raise` from the user. Every permit decision, allow or deny, is appended to `data/permit-audit.jsonl`.

If a permit check denies an action, the agent stops. No workaround. It reports what was attempted and what rule said no.

## How the substrate exposes itself

The work queue, permission engine, conformance audits, typed graph, weekly digest, and event-driven hooks are all agent-aware but can also be driven by hand. See the "Working with the substrate" section of the [README](../README.md#working-with-the-substrate) for the full command list.
